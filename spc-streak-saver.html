Case Study

SPC “Streak Saver” — tiny build, real lift

A lightweight feature that rewards students for maintaining a savings streak (redeem at least once per day/week). It nudges positive behavior and lifts offer engagement—aligned to SPC’s mission: Spend. Save. Repeat.

Context & Goals

  • Audience: Canadian students using SPC for daily savings.
  • Goal: Turn the slogan into a habit loop to improve recurring redemptions.
  • Scope: 1-week spike showing end-to-end ownership (UI, API, QA, analytics, CI).
Student-first Fast iteration Feature-flagged Partner-friendly

Solution Overview

Add a StreakCard on Home with progress + next reward. Redeeming an offer advances the streak; milestone rewards trigger. Events are tracked and A/B-testable (3-of-5 vs 7-day).

Feature Slice

  • StreakCard (home widget) with progress + reward preview
  • Offer list/detail → redeem flow
  • Rewards modal (milestones, confetti)
  • Analytics hooks + variant flags

Tech

  • React Native (Expo), React Query, AsyncStorage
  • Flask REST API (SQLite for spike), JWT stub
  • GitHub Actions (lint/tests), env-based flags
  • Amplitude/GA4 ready; CSV fallback logging

Architecture (minimal, scalable)

Data Model

users(id, created_at)
offers(id, partner_id, title, terms, points_value, geofence_json)
redemptions(id, user_id, offer_id, ts)
streaks(user_id, current_count, last_redeem_ts, best_count)

API Endpoints

GET  /offers
GET  /streak
POST /redeem  // advances streak; validates offer; emits events

Success Metrics

  • Activation: ≥60% of first-time viewers tap “Start my streak”.
  • Engagement: +20–30% lift in offer detail views (streak vs control).
  • Completion: ≥25% weekly streak completion.
  • Qualitative: Micro-survey “felt motivated to save” ≥4/5.

QA & Accessibility

  • Unit tests (UI + API) for happy/sad paths; double-redeem same day; offline cache.
  • Accessibility: labeled controls, focus order, ≥AA contrast, touch targets ≥44px.
  • Performance: Home render <500ms; list virtualization; API p95 <250ms on dev infra.

1-Week Timeline

  1. Day 1: KPIs, wires, repo + CI scaffold, event map.
  2. Day 2: Flask endpoints + seed data + auth stub.
  3. Day 3: RN components (StreakCard, Offer screens).
  4. Day 4: Tests + analytics hooks + A/B flag.
  5. Day 5: Edge cases, polish, demo recording.
